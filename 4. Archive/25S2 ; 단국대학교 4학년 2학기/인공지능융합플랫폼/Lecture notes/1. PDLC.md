플랫폼의 기획부터 폐기까지 효과적으로 관리하기 위한 순서를 정리한 것이다.


**계획 -> 분석 -> 설계 -> 구현 및 분석**


## 1단계: 계획 - 플랫폼을 왜 만들건데? 어디까지 만들건데?

- 플랫폼의 니즈 분석
- 플랫폼에 대한 잠재적 프로젝트 정의
- 그 플랫폼의 지속 여부를 결정

### 플랫폼 개발 계획 단계의 개발 프로세스

플랫폼을 막 만들기 전에 무엇을 만들지, 이게 진짜 가능한지, 어떻게 만들지를 정리하는 전 과정이다.

1. 플랫폼의 확인과 선정
	- 시장과 사용자 니즈를 분석해서 "우리가 만들 플랫폼이 새 서비스를 제공할 건지, 기존 플랫폼을 개선할 건지, 아니면 더 좋은 운영 환경을 주는 건지" 같은 큰 방향을 정하고, 경쟁 서비스 벤치마킹이나 시장조사를 통해 어떤 플랫폼을 할지 주제를 선택해.
2. 플랫폼의 초기화와 계획
	- 여기서는 플랫폼의 전체적인 비전과 목표를 정하고, 누가 개발에 참여할지 팀 구성을 하고, 어떤 개발 환경을 쓸지, 예산은 어느 정도가 필요한지, 타깃 사용자는 누구이고 어떤 관계를 맺을지 같은 기본 계획을 문서로 정리해 둔다.
3. 실현가능성(feasibility) 평가
	- 단순히 "좋은 아이디어다"에서 끝나는 게 아니라,
		- 경제적으로 비용과 예산이 감당 가능한지
		- 운영 측면에서 인력과 역량이 충분한지,
		- 기술적으로 구현이 가능한지,
		- 법적,계약적으로 문제가 없는지,
		- 전략적으로 회사 방향과 맞는지
	- 같은 요소들을 체크해서, 정말 실행 가능한 프로젝트인지 따져본다.
4. 기본 플랫폼 개발 계획 구축
	- 앞에서 정리한 초기 계획과 실현가능성 분석을 바탕으로, 플랫폼이 어떤 기능을 제공하고 어떤 구조로 움직일지, 그리고 전체 개발 일정과 마일스톤을 포함한 "개발 청사진"을 구체적으로 작성하는 단계라고 보면 돼.
5. 기본 플랫폼 개발 계획의 검토
	- 마지막으로 개본 플랫폼 개발 계획의 검토가 이루어진다. 내부 회의를 통해 지금까지 세운 계획과 분석에 빠진 부분이 없는지, 리스크는 없는지, 목표와 일정이 현실적인지 다시 한 번 점검하고 보완해서 "이대로 개발을 시작해도 된다"는 수준까지 계획을 다듬는 걸로 계획 단계가 마무리된다.



### 계획 단계에서의 두 가지 주요 활동

1. "기존 플랫폼보다 더 나은 플랫폼인가? / 어디에서도 제공되지 않는 기능을 제공하는가?"를 체크해야한다.
2. 제안된 플랫폼의 서비스 제공 범위를 결정해야한다.

-> 주요 산출물: 플랫폼 프로젝트 제안서(배경, 목표, 계획, 범위, 유사 사례)


## 2단계: 분석 - 정확히 무엇을 만들어야 하지?

2단계는 요구사항을 캐내고 정리하는 단계이다.
계획 단계가 "이거 해보자"였다면, 
분석 단계는 "그래서 정확히 뭘 해줘야 하는데?"를 파고드는 과정이라고 보면 돼.

### 먼저 플랫폼 요구사항 조사를 한다.
- 플랫폼의 목표
- 어떤 정보와 데이터가 오가는지
- 관련 비즈니스/시장/정책 환경은 어떤지
이런 것들을 폭넓게 수집한다.

여기서 핵심은 사용자가 정말 원하는 것과
이미 존재하는 유사 제품･정책･규제를 파악하는 거야.
그래서 보통
- 개인 인터뷰 -> **Gradi 영업할 때, 인터뷰로 접근하는 것도 괜찮은 방법인듯??**
- 실제 작업자 관찰
- 기존 문서/보고서 분석
같은 전통적인 방법을 쓴다고 나와 있다.

이렇게 모은 정보를 바탕으로, 
우리는 요구사항을 "정의"하는 단계로 들어간다.
여기서 정의해아 할 것들은 예를 들면 이런 것들이다.

- 기존 플랫폼의 문제점과 개선 아이디어
- 관련 법규/규정(온라인 상거래법, 저작권법)
- 최신 기술 동향과 우리가 쓸 기술
- 타깃 사용자, 시장 현황과 전망
- 플랫폼의 배경･목적, 구성 기술, 주요 서비스 리스트
- 비즈니스 모델, 개발 계획서 등

쉽게 말해,
이 플랫폼이 어떤 환경에서, 누구에게, 어떤 기능과 품질로 서비스해야 하는가를
문서로 꽉 잡아 놓는 단계가 2단계야.
이게 탄탄해야 3단계 설계에서 해매지 않는다.

## 3단계: 설계 - 그러면 어떻게 만들어야 하지?
플랫폼의 뼈대, 모양, 데이터를 설계하는 단계

2단계 분석이 "무엇을 만들지"를 명확히 했다면,
3단계 설계는 "그걸 어떻게 구현할지"를 구조적으로 그리는 단계다.
즉, UI부터 데이터베이스, 모듈 구조까지 플랫폼의 모든 설계도를 만드는 과정이다.

### 설계 단계의 핵심 개념
- 논리적 설계
	- 플랫폼의 기능과 데이터 흐름을 기계/서버 같은 특정 환경과 상관없이 "개념적, 구조적으로" 설계하는 단계

- 물리적 설계
	- 실제 개발 환경, DB 구조, UI 배치, 네트워크 구조 등을 기술적으로 설계하는 단계.

### 설계 단계의 주요 프로세스

1. **상황도**(플랫폼 전체 개요) 작성 - 플랫폼이 어떤 범위와 환경 속에서 작동하는지 표현
	- 플랫폼 경계(사용자/관리자/외부 API 등)와 단 하나의 시스템(프로세스) 구조를 단순하게 표현한다.
	- 아직 내부 기능은 넣지 않고, 이해당사자 - 시스템 간 데이터 흐름만 그린다.

2. 서비스 기능을 모듈 단위로 구조화(모듈화)
	- 분석 단계에서 나온 기능 목록을 정리하고, 중복 기능을 제거하고, 비슷한 기능을 그룹으로 묶어 모듈로 만든다.
	- "숙소 관리 / 회원 관리 / 추천 / 분석"과 같이 서로 독립적인 기능 단위로 나누는 작업이다.

3. 데이터 흐름도(서비스 모델) 작성 - 서비스가 실제로 어떻게 동작하는지 흐름 그리기
	- 상황도에서의 큰 시스템을 여러 프로세스로 분해
	- 프로세스 간 데이터 이동을 구체적으로 표현
	- 사용자가 서비스를 사용할 때 어떤 데이터가 움직이는지 한눈에 보이도록 구성
		-> 이 단계까지가 논리적 설계(Logical)


### UI 설계 - 사용자가 실제로 보게 될 화면 구조 만들기 (물리적 설계 part 1)



![[Pasted image 20251201135544.png]]

4. 대화도(Dialog Diagram) 작성
	- 플랫폼 내 모든 화면(페이지)을 "스토리 흐름" 형태로 배치
	- 각 화면이 어떤 화면에서 왔는지, 어디로 이동하는지 명확히 보이게 함
		-> 예: 로그인 -> 메인 -> 마이페이지 -> 설정

5. 프로토타이핑(Wireframe / Mockup) 작업
	- 실제 화면 배치, 버튼, 입력폼, 리스트 등 UI 구조를 시각적으로 설계
	- 카카오 Oven, Figma, Mockup 등 툴로 구현
	- 고려해야할 점:
		- 색/구조/아이콘의 일관성
		- 사용자가 지금 무엇을 하고 있는지 명확히 보이도록
		- 긴 작업은 로딩/진행률 표시 등 사용자 피드백 제공


### DB 설계 - 서비스 데이터를 어떻게 저장하고 연결할지 결정 (물리적 설계 part 2)

6. 논리적 DB 설계
	- 데이터 테이블과 그 속성(필드) 목록을 정의
	- 중복을 제거하기 위한 정규화 과정 수행

7. 물리적 DB 설계
	- 각 속성의 Data Type, 저장 방식, 인덱스 구조 등을 기술적으로 설정
	- 실제 서버,클라우드 환경을 고려해 최적화

8. ERD(Entity-Relationship Diagram)작성 - 최종 데이터베이스 구조 완성
	- 모든 테이블(PK), 관계(FK), 제약조건 등을 시각적으로 표현
	- 실제 개발에서 바로 사용할 수 있는 최종 DB 구조


### 9. 플랫폼 구조도 작성 - 플랫폼 전체 구조를 한 장에 정리하는 최종 설계 작업

설계 단계의 하이라이트
1~3단계의 모든 내용을 종합해 플랫폼 전체 구조를 한눈에 보여주는 도식을 만든다.

- 모듈 구성
- 모듈 간 상호관계
- 외부 API, 타사서비스, 인프라 구조
- 사용자/관리자/시스템 간 데이터 흐름
- 플랫폼의 경계 표시

**중요 포인트**
- 구성요소 위치와 관계가 논리적으로 맞는가?
- 모듈 구성이 중복 없이 정리됐는가?
- 외부 환경(API/플랫폼)과의 관계가 명확한가?
- 경계(플랫폼 vs 외부)가 구분되어 있는가?


## 4단계: 구현 및 운영 - 이제 실제로 만들고 돌려보는 단계

설계가 완성되면, 그 설계서를 기반으로 실제로 코드를 작성하고, 테스트하고, 설치하고, 운영과 개선까지 진행하는 단계다.
플랫폼이 실제 제품으로 태어나는 단계이자, 이후 지속적으로 개선되는 주기적 단계다.


### 구현 및 운영 단계의 주요 활동

1. 코딩(개발) - 설계한 대로 기능을 실제로 만든다.
	- 모듈별로 프로그램 구현
	- UI, 서버, DB, API 연동 등 전체 기능을 개발

2. 테스트 - 오류를 잡고 품질을 올린다.
	- 단위 테스트 -> 통합 테스트 -> 시스템 테스트
	- 설계와 요구사항에 맞게 동작하는지 검증
	- 발견된 오류는 수정 후 재테스트

3. 설치(Deployment) - 사용자 환경에 배포
	- 사용자용(앱/웹 등) 설치 및 배포
	- 관리자/운영자 시스템 설치(서버 세팅 등)

---

### 기타 필수 운영 활동

4. 문서화 - 모든 산출물 마무리
	- 개발 문서, 매뉴얼, 최종 기술 보고서 작성
	- 향후 운영/유지보수를 위한 문서 정리

5. 교육 - 운영 인력/ 관리자 교육
	- 시스템 사용법
	- 장애 대응법
	- 운영 절차 등

6. 지원(Support) - 사용자/운영자 지원
	- 고객 문의 대응
	- 장애 처리
	- 셧다운, 업데이트 등 운영 핸들링

### 7. 유지보수 - 운영 과정에서 발생하는 문제를 지속적으로 개선

- 사용자 요구 변화 반영
- 버그 수정
- 성능 개선
- 새로운 정책, 기술, 환경 변화 적용
	-> PDLC는 순환 구조이기 때문에, 유지보수 과정에서 다시 1단계(계획)로 돌아갈 수 있다.
