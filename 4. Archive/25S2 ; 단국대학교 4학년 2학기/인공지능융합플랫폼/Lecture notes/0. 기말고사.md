	- 리서치를 통해 어떤 플랫폼을 할지 주제를 선택

### 2) 분석
계획을 구체적으로 분석 -> 그래서 뭘 하는데?

- 사용자가 정말 원하는게 뭘까?

## 5. 빅데이터

### 개요
- 방대한 양의 데이터
- 대량의 정형 / 비정형 데이터를 분석해 가치를 창출


### 과거와 현재의 데이터 차이

**과거**
- 데이터 형태: 특정 양식에 맞춰 분류
- 데이터 속도: 배치(Batch) -> 느림
- 데이터 처리 목적: 과거를 분석
- 데이터 처리 비용: 국가 정부에서 부담해야할 수준

**현재**
- 데이터 형태: 형식이 없고 다양함
- 데이터 속도: 거의 실시간
- 데이터 처리 목적: 최적화 혹은 예측
- 데이터 처리 비용: 개별 기업에서 부담할 수 있는 수준

### 빅데이터 특징 - 6V
1. Volume - 크기
2. Variety - 다양성
3. Velocity - 속도

4. Veracity - 정확성(쓸만 해야함)
5. Variability - 가변성(맥락에 따라 변함)
6. Visualization - 시각화

### 빅데이터 플랫폼
- 빅데이터 기술을 잘 사용할 수 있도록 준비된 환경
- 빅데이터 분석 및 활용을 위한 필수 인프라

-  데이터 발생 -> (데이터 수집 -> 분석 / 처리 / 저장 관리 -> 시각화) -> 서비스 제공


### 빅데이터 분석
1. 분석 기술
	- 기본적으로 Divide & Conquer -> 데이터를 독립된 형태로 나누고 병렬 처리
		- 아파치 하둡: Map Reduce
		- NoSQL: 예를 들어 Cassandra

2. 표현 기술
	- 분석 결과를 시각화
	- R, Python 등

3. 분석 종류
	- 현황 분석
		- 상황 파악 및 진단 / 진단 결과를 알아보기 쉽게 정리 / 사건에 대한 원인 파악

	- 예측 분석

	- 예측 최적화 분석
		- 예측을 가장 잘 할 수 있는, 최적화된 모델을 찾기 위한 분석

### 빅데이터 활용 사례
- 정치
	- 미국 대선 -> 유권자 맞춤형 선거 전략
- 경제 및 경영
	- 아마존 닷컴의 추천상품 표시
- 문화
	- MLB의 머니볼 이론(오직 데이터를 기반으로 선수들을 배치해서 승률을 높인다는 게임 이론)
- 과학기술
	- 단백질 서열 분석

### 빅데이터 플랫폼 기본 구성

데이터 영역  /  하둡  /  전사 기업 정보 관리  /  분석 툴, 인사이트, BI  /  데이터 시각화


**빅데이터 플랫폼 구축 과정**
데이터 소스 -> ETL -> 신호 생성 -> 분석 -> 접근 -> 조치

ETL: Extract, Transform, Load



## 문제 풀이

## **🔥 핵심을 묻는 문제 3개**

### **문항 1 — PDLC와 플랫폼 설계 통합 사고**

  

플랫폼 서비스를 새로 기획한다고 할 때,

**PDLC의 ‘계획–분석–설계’ 단계 각각에서 반드시 수행해야 할 핵심 활동을 플랫폼 아키텍처의 주요 요소(Users, UI, Services, Data Storage 등)와 연결하여 설명하라.**

특히 _사용자 요구 파악 → 논리적 설계 → 물리적 DB 설계_가 어떻게 일관된 흐름을 이루는지 서술하라.

---

### **문항 2 — 빅데이터 6V와 플랫폼 구조**

  

어떤 플랫폼이 실시간 추천 서비스를 제공하기 위해 빅데이터 기술을 도입하려 한다.

이때 **빅데이터의 6V 특징이 플랫폼 아키텍처의 어떤 구성 요소(수집, 저장, 분석, 시각화, 서비스, 보안 등)에 가장 강하게 영향을 미치는지**를 분석하고,

또 **Velocity와 Variety가 플랫폼 구조 설계에 주는 구체적 제약 조건**을 설명하라.

---

### **문항 3 — 분석 단계와 빅데이터 분석 종류의 연결**

  

PDLC의 ‘분석 단계’에서 사용자 요구 조사 후 플랫폼 개선 방향을 도출해야 한다.

이를 위해 **현황 분석·예측 분석·예측 최적화 분석** 중 어떤 분석을 어떤 순서로 적용해야 하는지 제안하고,

각 분석이 **플랫폼 가치 요소(핵심 제공 가치·수익 공식·핵심 자원·핵심 프로세스)** 중 무엇을 강화하는지 논리적으로 설명하라.

# **✅** 

## **문항 1 채점: 7.5 / 10**

### **👍** 

### **좋았던 점**

- PDLC의 각 단계에서 User·Service·UI·DB 등을 고려해야 한다는 흐름을 잘 잡음.
    
- “계획 단계에서 User, Service 선정”, “설계 단계에서 UI, DB 설계” 등 기본 구조는 정확함.
    

  

### **❗ 보완이 필요한 점**

1. **계획 단계에서 Service/User 정의는 ‘방향 설정’ 수준이지 구체화 수준이 아님**
    
    → 계획 단계는 _테마/플랫폼 영역 선정_ + _타깃 사용자 범주 파악_ 정도임.
    
    분석 단계에서 **구체적인 사용자 니즈·행동·문제 정의**가 이루어져야 한다.
    
2. **설계 단계 설명이 부족함**
    
    → 논리적 설계와 물리적 설계의 차이를 구체적으로 언급해야 함.
    
    → 서비스 서버의 구조뿐 아니라 _모듈화, 데이터 흐름도, 서비스 모델_도 포함됨.
    

---

## **⭐** 

## **문항 1 모범답안**

  

계획 단계에서는 **플랫폼의 방향성과 범위를 결정하며**, 이를 위해 타깃 사용자군(Users)과 제공할 가치(Services)의 대략적 카테고리를 설정한다. 이 단계에서는 구체화보다 “무엇을 위한 플랫폼인가”를 정의한다.

  

분석 단계에서는 계획에서 세운 가설을 **사용자 조사, 경쟁 분석, 정책 검토 등을 통해 구체화**하며, Users·Services의 실제 니즈, 문제점, 행동 패턴을 정밀하게 파악한다. 분석으로 얻은 결과는 이후 설계의 요구사항이 된다.

  

설계 단계에서는 분석된 요구사항을 기반으로 **논리적 설계(상황도, 서비스 흐름도, 모듈 구조)**와 **물리적 설계(UI/Prototype, DB 구조, 서버 아키텍처)**를 작성한다. UI는 사용자 인터페이스 흐름을 시각적으로 표현하며, DB·서비스 서버 구조는 기능을 실현할 수 있도록 기술적으로 구체화된다.

# **✅** 

# **문항 2 채점: 5.5 / 10**

  

### **👍** 

### **좋았던 점**

- Volume → 수집, Visualization → 서비스 영향 등 큰 틀은 나쁘지 않음.
    

  

### **❗ 보완이 필요한 점**

1. **Veracity의 해석이 틀림**
    
    → “정확성 때문에 좋은 데이터만 저장해야 한다”는 설명은 부족함.
    
    실제로 Veracity는 **불확실성이 높은 데이터를 어떻게 정제·보정할지**의 문제임.
    
    단순히 저장 단계 문제가 아니라 **수집–정제–모델링 전체에 영향**을 줌.
    
2. **Velocity와 Variety 관련 요구를 제대로 반영하지 못함**
    
    문제는 Velocity와 Variety가 **설계 단계의 제약 조건**에 어떤 영향을 미치는지 구체적으로 설명하라고 했음.
    
    → 실시간 처리(스트리밍 구조 필요), 캐싱 전략, 스케일 아웃 구조, NoSQL 선택 등의 설계적 영향이 필요했음.
    
3. Visualization이 서비스에만 영향을 미친다고 보기에는 협소함
    
    → Visualization은 **분석·모델링 결과를 사용자나 관리자에게 전달하기 위한 계층**이며 서비스 UI 뿐 아니라 내부 BI 시스템에도 영향.

## **⭐** 

## **문항 2 모범답안**

  

Volume은 방대한 데이터를 처리해야 하므로 **수집 시스템과 저장소의 확장성 설계(분산 파일 시스템, 샤딩, 스케일 아웃)**에 직접적인 영향을 준다.

  

Variety는 정형·비정형 데이터의 다양성 때문에 **데이터 스키마 설계, NoSQL 도입, ETL/ELT 파이프라인 처리 방식**에 영향을 준다.

  

Velocity는 실시간 데이터 처리 요구 때문에 **스트리밍 처리 엔진(Kafka, Spark Streaming), 메모리 기반 처리, 실시간 API 구조**를 플랫폼 설계에 반영하게 만든다.

  

Veracity는 데이터의 불확실성을 해결하기 위한 **정제·검증 프로세스, 데이터 품질 관리 모듈, 로그 신뢰도 처리 구조**를 필요로 한다.

  

Visualization은 분석 결과를 직관적으로 전달하기 위해 **대시보드, 리포팅 시스템, 시각화 엔진**을 서비스 및 관리자 도구에 통합시키는 데 영향을 준다.

# **✅** 

# **문항 3 채점: 8.5 / 10**

  

### **👍** 

### **잘한 점**

- 분석 순서를 “현황 → 예측 → 예측 최적화”로 진행한 것은 매우 합리적.
    
- PDLC 분석 단계의 역할과 빅데이터 분석 종류를 자연스럽게 연결함.
    
- 전체적으로 논리적 흐름이 우수함.
    

  

### **❗ 보완이 필요한 점**

- 각 분석이 **플랫폼 가치 요소(핵심 제공 가치, 수익 공식, 핵심 자원, 핵심 프로세스)**와 어떻게 연결되는지 언급이 부족함.
    
- 예측 분석이 “개선 방향 적용 후 미래 예측”뿐만 아니라 **타깃 사용자 반응, 시스템 부하, 비용 구조 변화**를 예측한다는 점까지 확장할 수 있음.

## **⭐** 

## **문항 3 모범답안**

  

먼저 **현황 분석**을 통해 플랫폼의 현재 문제, 사용자 행동 패턴, 병목 구간을 파악한다. 이를 통해 핵심 제공 가치와 핵심 프로세스의 개선 포인트가 도출된다.

  

다음으로 **예측 분석**을 실시하여 새로운 기능 도입이나 정책 변화가 사용자 행동·수익·운영 비용에 어떤 영향을 줄지 시뮬레이션한다. 이는 수익 공식과 핵심 자원의 효율성을 평가하는 데 중요한 역할을 한다.

  

마지막으로 **예측 최적화 분석**을 적용하여 여러 개선안 중 최적의 조건을 찾고, 가장 효율적인 기능 배치·추천 모델·로드 밸런싱 전략 등을 도출한다. 이는 플랫폼의 핵심 프로세스를 최적화하고 전략적 의사결정의 근거가 된다.
